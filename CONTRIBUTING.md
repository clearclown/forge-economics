# Contributing to Forge Economics

## はじめに

Forge Economics へようこそ。本リポジトリは、Forge プロトコルの経済理論 (economic theory) を体系的にまとめたドキュメント集です。従来の経済学の枠組みから出発し、Compute Unit (CU) を基軸とした新しい経済モデルを提示することを目的としています。

以下のような貢献を歓迎します:

- 経済理論そのものに関する議論・修正提案
- 文章の校正、誤字脱字の修正、読みやすさの改善
- 図解 (ASCII art / Mermaid) の追加・改善
- 具体例・ケーススタディの追加
- 翻訳 (日本語 → 他言語)
- Open Question (Chapter 13) への回答・深掘り
- SPEC-AUDIT で指摘された issue の解決

初めての方は、まず `README.md` と `SPEC-AUDIT.md` に目を通してから、小さな修正 PR (typo 修正など) から始めることをおすすめします。

## 執筆規約

### 言語

- 本文は **日本語** で記述します。
- 技術用語 (technical term) は **初出時に英語併記** してください。例: 「計算単位 (Compute Unit, CU)」「動的価格 (dynamic pricing)」。
- 二回目以降は日本語または略語のみで構いません。

### 章構造

各章は原則として以下の **三部構成** で書いてください:

1. **従来の経済学では** — 既存の経済理論ではこのトピックがどう扱われているか
2. **Forge ではどうなるか** — CU 経済では同じトピックがどう扱われるか
3. **なぜ違うのか** — 両者の差異の根本原因 (物理的制約・インセンティブ設計・技術的前提の違い)

この構造により、読者は既存知識との差分として Forge の主張を理解できます。

### 図解

- **ASCII art** と **Mermaid** を併用してください。
- 簡単なフロー・階層図は ASCII art で、複雑な状態遷移やシーケンスは Mermaid で表現します。
- 図解は必ず本文中の該当箇所の近くに配置し、キャプションで何を示しているかを明記します。

### 数式

- 数式は **最小限** に留めてください。
- 代わりに **具体例** (数値を入れた計算例) を優先します。
- どうしても数式が必要な場合は、LaTeX ではなくインライン表記 (例: `P = D / S`) を使い、続けて日本語の説明を添えます。

## 用語の統一ルール

| 用語 | 表記ルール |
|------|-----------|
| Compute Unit | 初出時「Compute Unit (CU)」、以降は **CU** と省略 |
| 1 CU の定義 | **1 CU = 10 億 FLOP** (すべてのドキュメントで統一) |
| welcome loan の数値 | 直接数値を書かず、**`spec/parameters.md` を参照** すること |
| Proof of Useful Work | 初出時「Proof of Useful Work (PoUW)」、以降は **PoUW** |
| 動的価格 | 初出時「動的価格 (dynamic pricing)」 |

新しい略語・用語を導入する場合は、PR 内で他章への影響を確認し、必要なら `docs/glossary.md` (存在する場合) も更新してください。

## PR の出し方

### PR の分離

**経済理論の修正** と **文章校正** は必ず別 PR にしてください。理由は以下のとおりです:

- 理論修正はレビュー観点が「主張の正当性」であり、議論が長期化しやすい
- 文章校正は「読みやすさ」が観点で、迅速にマージすべき
- 両者を混ぜるとレビューが停滞する

### SPEC-AUDIT.md の更新

新しい issue を追加・更新する場合:

1. `SPEC-AUDIT.md` に新しい行を追加 (ID, 重要度, 該当章, 概要)
2. ID の採番規則は既存エントリに従う (例: `M-6`, `H-3`)
3. 対応する GitHub Issue を作成し、SPEC-AUDIT.md からリンク
4. PR 本文に「SPEC-AUDIT M-x に対応」と明記

### verify-audit.sh

PR を出す前に `scripts/verify-audit.sh` (存在する場合) を実行し、その **出力を PR 本文に貼り付けて** ください。これによりレビュアーは整合性チェックの結果を即座に確認できます。

```bash
./scripts/verify-audit.sh
```

## Issue ガイド

Issue は以下の 3 種類にラベル分類してください:

| ラベル | 用途 |
|--------|------|
| `bug` | 誤記・矛盾・リンク切れなど、明確な誤り |
| `discussion` | 解釈の議論、理論的な論点 |
| `proposal` | 新しい章・節の追加、構成変更の提案 |

### Open Question (Chapter 13)

Chapter 13 「Open Questions」に関連する issue は、タイトル先頭に **`[Open Question]`** プレフィックスを付けてください。

例: `[Open Question] CU の貯蓄と流通速度のトレードオフ`

これにより、未解決の理論的問いをフィルタリングできます。

## レビュープロセス

レビュアーは以下の観点で PR を確認します:

### 1. 経済学的な主張の正当性

- 前提 (assumption) が明示されているか
- 既存の経済学との対比が公平か (従来理論を藁人形化していないか)
- 反証可能性 (falsifiability) があるか
- 具体例が主張を支えているか

### 2. 章間の用語整合性

- 用語が「用語の統一ルール」に従っているか
- 他章で異なる定義・異なる数値が使われていないか
- 初出時の英語併記が漏れていないか

### 3. 数値・パラメータの整合性

- `spec/parameters.md` と矛盾する数値が本文にハードコードされていないか
- 数値を変更した場合、すべての参照箇所が更新されているか (後述)

レビューは最低 1 名、理論的影響の大きい PR は 2 名以上の承認を推奨します。

## 数値定数の変更時の注意

**数値定数のシングルソース (single source of truth) は `spec/parameters.md` です。**

### 手順

1. `spec/parameters.md` を更新する
2. すべての章で該当数値を `grep` し、整合性を確認する
   ```bash
   grep -rn "1000" docs/   # 例: welcome loan の旧値を探す
   grep -rn "10億 FLOP" docs/
   ```
3. 本文中で数値をハードコードしている箇所があれば、可能な限り「詳細は `spec/parameters.md` 参照」という記述に置き換える
4. 変更理由と影響範囲を PR 本文に記載する
5. `scripts/verify-audit.sh` を実行して整合性が取れていることを確認する

### やってはいけないこと

- `spec/parameters.md` を更新せずに本文の数値だけ変更する
- 一部の章だけ新しい数値に更新し、他章を放置する
- 数値変更を文章校正 PR に混ぜる (必ず独立した PR にする)

## 質問・相談

不明な点があれば、気軽に Issue で質問してください。`discussion` ラベルを付けて投稿すれば、理論的な議論の場としても活用できます。

Forge Economics は、AI と人間が共に育てる経済理論のドキュメントです。皆さんの貢献をお待ちしています。
